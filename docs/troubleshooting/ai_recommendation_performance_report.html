1.개요

본 보고서는 P-Tuk-Tuk 프로젝트의 AI 기반 분리수거 물품 추천 서비스의 성능 개선 과정을 정리한 문서입니다. Gemini AI 모델 변경(gemini-2.5-flash → gemini-2.0-flash-lite), 프롬프트 최적화, GenerationConfig 튜닝을 통해 평균 응답 시간을 약 20초에서 1초 이내로 단축하여 실용적인 서비스로 전환하였습니다.

핵심 성과: 평균 응답 시간 20,716ms → 1,000ms (95.2% 개선)
특이사항: 워밍업 효과로 연속 요청 시 649ms까지 단축

2.문제 정의

2.1 초기 구현의 성능 문제

초기 구현 코드:

public Optional<AIRecommandDto> getRecommand(String itemName) { try (VertexAI vertextAI = new VertexAI(projectId, location)) { /** * Item Keyword (Redis Cache) * Keyword Specific (Spring cache) * * 리스트 Setting */ String prompt = String.format(PROMPT_TEMPLATE, itemName); // ❌ 무거운 모델 사용 GenerativeModel model = new GenerativeModel( "gemini-2.5-flash", vertextAI ); // ❌ 설정 없이 기본값 사용 GenerateContentResponse response = model.generateContent(prompt); } }

  • gemini-2.5-flash 모델의 과도한 처리 시간

3.해결 방안

3.1 AI 모델 변경 (핵심 개선 - 약 60% 기여)

gemini-2.5-flash → gemini-2.0-flash-lite 변경

구분 gemini-2.5-flash gemini-2.0-flash-lite
모델 크기 중형 멀티모달 모델 경량 텍스트 특화
응답 속도 느림 (~20초) 빠름 (~1초)
토큰 비용 높음 낮음

3.2 GenerationConfig 튜닝 (약 25% 기여)

GenerationConfig config = GenerationConfig.newBuilder() .setTemperature(0.2f) // 일관된 출력 .setTopP(0.9f) // 상위 90% 토큰 .setTopK(10) // 후보 10개 제한 .build();

효과: 토큰 생성 단계 축소 → 응답 속도 25% 향상

3.3 프롬프트 최적화 (약 15% 기여)

프롬프트 토큰 수: 500 tokens → 100 tokens (80% 감소)

5.워밍업 효과 분석

5.1 왜 재사용될수록 빨라지는가?

워밍업 효과의 원인:

  • JIT 컴파일러 최적화: 반복 실행되는 코드 경로가 기계어로 컴파일됨
  • 클래스 로딩 완료: 필요한 클래스가 모두 메모리에 로드됨
  • HTTP 연결 풀 재사용: VertexAI API 연결이 유지됨
  • CPU 캐시 효율: 자주 사용하는 데이터가 L1/L2 캐시에 상주

5.2 응답 시간 감소 패턴

1차 → 2차: 2,940ms → 1,818ms (38% 감소)

  • 주요 클래스 로딩 및 JIT 컴파일 시작

2차 → 6차: 1,818ms → 674ms (63% 감소)

  • 핫스팟 최적화 진행 및 연결 풀 안정화

7차 이후: 649~850ms (안정화)

  • 최적화 완료 상태 유지

5.3 실제 서비스 시나리오별 성능

6.트러블슈팅 및 학습 내용

6.1 싱글톤 패턴의 오해

초기 가설: VertexAI를 매번 초기화하는 것이 문제라고 생각했으나, 실제로는 싱글톤 적용 후에도 큰 성능 향상이 없었습니다.

실제 원인: AI 모델 자체의 처리 시간이 병목이었습니다. gemini-2.5-flash가 너무 무거웠던 것이 주요 원인입니다.

6.2 AI 모델 선택의 중요성

핵심 교훈:

  • 최신/고성능 모델 ≠ 최선의 선택
  • 작업 복잡도에 맞는 모델 선택 중요
  • 단순 분류는 lite 모델이 최적

7.결론 및 향후 계획

7.1 결론

AI 모델을 gemini-2.0-flash-lite로 변경하고, GenerationConfig 튜닝 및 프롬프트 최적화를 통해 응답 시간을 95% 이상 단축했습니다. 특히 워밍업 효과로 연속 요청 시 649ms까지 단축되어 실시간 서비스가 가능해졌습니다.

핵심 성과: 평균 20,716ms → 1,000ms (95.2% 개선)

7.2 향후 개선 계획

  • Redis 캐싱: 동일 키워드 재요청 시 AI 호출 생략 → 0ms 목표
  • 배치 처리: 여러 상품 한 번에 처리
  • Aho-Corasick 알고리즘: 키워드 매칭 고도화

부록 A. 최종 구현 코드

@Slf4j @Service public class AiRecommendationService { @Value("${gcp.project.id}") private String projectId; private VertexAI vertexAI; @PostConstruct private synchronized void ensureInitialized() { if (this.vertexAI == null) { this.vertexAI = new VertexAI(projectId, location); } } public List<Item> recommend(String keywords) { long start = System.currentTimeMillis(); // ✅ 최적화된 설정 GenerationConfig config = GenerationConfig.newBuilder() .setTemperature(0.2f) .setTopP(0.9f) .setTopK(10) .build(); // ✅ 경량 모델 사용 GenerativeModel model = new GenerativeModel( "gemini-2.0-flash-lite", vertexAI ).withGenerationConfig(config); String prompt = buildOptimizedPrompt(keywords); GenerateContentResponse response = model.generateContent(prompt); long duration = System.currentTimeMillis() - start; log.info("Vertex AI 응답 시간: {}ms", duration); return parseResponse(response); } }
